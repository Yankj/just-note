# 产品战略：极简 AI 知识记录工具

**创建时间**: 2026-03-26  
**最后更新**: 2026-03-26  
**状态**: 战略思考阶段

---

## 一、核心洞察

### 1.1 灵感来源

- **触发点**: 微信公众号文章《ima 调研》
- **核心痛点**: 「打开 Notion 界面、操作一通之后想记录的灵感已经忘了」
- **关键观察**: IMA 接入 AI 能力，但本质仍是「文档内整理、知识库检索」，输入门槛依然高

### 1.2 产品哲学

> **从输入开始就要足够简单**

| 传统工具 (Notion/IMA) | 本产品理念 |
|---------------------|-----------|
| 打开应用 → 新建文档 → 选择模板 → 编辑 | 直接输入（消息/语音） |
| 用户手动分类、打标签 | AI 自动归类、关联 |
| 用户设计文档结构 | AI 自动排版、组织 |
| 以「文档」为中心 | 以「想法」为中心 |

### 1.3 一句话定位

> **像发微信一样记录灵感，AI 自动帮你整理成知识库。**

## 二、技术路线分析

### 2.1 两种方案对比

#### 方案 A: OpenClaw + Skill（文件/数据库持久化）

```
微信/飞书 → OpenClaw Gateway → 笔记 Skill → 文件/数据库
```

**优势**:
- ✅ 开发成本低（1-2 周 MVP）
- ✅ 复用现有通道（微信/飞书已集成）
- ✅ 文件天然可读、可 Git 版本控制
- ✅ 适合极客用户

**劣势**:
- ⚠️ 检索性能依赖额外索引
- ⚠️ 普通用户上手有门槛
- ⚠️ 功能受 OpenClaw 边界限制

#### 方案 B: 独立 App（数据库 + AI 处理）

```
微信/飞书 → 独立后端 API → AI 处理 → 数据库
```

**优势**:
- ✅ 完整产品体验（专用检索/管理界面）
- ✅ 性能可控（数据库索引、水平扩展）
- ✅ 适合大众用户（90%+ 市场）
- ✅ 独立产品，可商业化

**劣势**:
- ⚠️ 开发成本高（1-2 月 MVP）
- ⚠️ 需自建消息通道对接
- ⚠️ 运维复杂度高

### 2.2 持久化方式对比

| 维度 | 文件持久化 | 数据库持久化 |
|------|-----------|-------------|
| 写入速度 | ✅ 快 | ✅ 快 |
| 检索速度 | ⚠️ 慢（需索引） | ✅ 快（原生索引） |
| 人类可读性 | ✅ 直接用文本编辑器 | ❌ 需专用工具 |
| AI 理解难度 | ✅ 低（Markdown 语义化） | ⚠️ 高（需理解 schema） |
| 版本控制 | ✅ 天然支持 Git | ❌ 需额外设计 |
| 并发写入 | ❌ 文件锁问题 | ✅ 事务支持 |

**结论**: 对于「灵感记录」场景（写多读少），**文件持久化更优**，但可加数据库索引层加速检索。

## 三、人群分层与产品定位

### 3.1 目标用户分层

| 人群 | 特征 | 占比 | 适合方案 |
|------|------|------|---------|
| **极客/开发者** | 熟悉命令行、愿意折腾、重视数据主权 | ~5-10% | OpenClaw + Skill |
| **普通用户** | 只想简单记录、不愿配置、习惯 SaaS | ~90%+ | 独立 App |

### 3.2 双产品战略

| 产品 | 定位 | 目标人群 | 运营模式 |
|------|------|---------|---------|
| **OpenClaw Skill** | 极客版/验证版 | 开发者、技术爱好者 | 开源 + 社区维护 |
| **独立 App** | 大众版/商业化版 | 普通知识工作者 | SaaS 订阅 |

**协同效应**:
- OpenClaw 验证核心假设 → 降低独立 App 风险
- OpenClaw 用户可迁移到独立 App → 早期种子用户
- 技术逻辑可复用 → 降低开发成本

## 四、分阶段执行计划

### 阶段 1: OpenClaw MVP 验证（1-2 周）

**目标**: 验证核心假设 —— 「用户是否愿意用消息方式记录灵感」

**投入**: 100% 精力

**关键任务**:
- [ ] 创建笔记 Skill（输入处理 + 文件写入 + 基础检索）
- [ ] 配置微信/飞书接入
- [ ] 自己深度使用 1 周
- [ ] 找 3-5 个极客朋友试用

**成功标准**:
- ✅ 每天主动记录 ≥ 3 次
- ✅ 每周主动检索 ≥ 2 次
- ✅ 朋友愿意持续使用

---

### 阶段 2: OpenClaw 迭代 + 独立 App 预研（2-4 周）

**目标**: 优化体验，同时为独立 App 做技术准备

**投入**: OpenClaw 80%, 独立 App 20%

**OpenClaw 线**:
- [ ] 根据反馈优化 AI 分类准确性
- [ ] 增加高级检索（标签组合、时间范围）
- [ ] 增加导出功能（Obsidian/Notion 格式）

**独立 App 线**:
- [ ] 技术选型（后端/前端/数据库）
- [ ] 架构设计
- [ ] UI/UX 原型设计

**关键决策点**（第 4 周末）:
```
如果 OpenClaw 版本日活 < 10 → 重新思考产品方向
如果 OpenClaw 版本日活 10-50 → 继续迭代，暂缓独立 App
如果 OpenClaw 版本日活 > 50 → 启动独立 App 开发
```

---

### 阶段 3: 独立 App MVP（1-2 月）

**触发条件**: OpenClaw 版本验证成功（日活 > 50）

**目标**: 服务大众用户

**模块规划**:
| 模块 | 技术选型 | 工时 |
|------|---------|------|
| 后端 | Node.js + FastAPI + PostgreSQL | 2 周 |
| 前端 | React + Tailwind | 2 周 |
| 微信接入 | 企业微信/第三方服务 | 1 周 |
| AI 处理 | 复用 OpenClaw 逻辑 | 3 天 |
| 部署 | Docker + 云服务器 | 3 天 |

---

### 阶段 4: 双产品运营（持续）

| 产品 | 定位 | 运营模式 |
|------|------|---------|
| OpenClaw Skill | 极客版 | 开源 + 社区维护 |
| 独立 App | 大众版 | SaaS 订阅/免费 + 增值 |

## 五、风险与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| OpenClaw 版本没人用 | 中 | 低 | 损失小（1-2 周），快速转向 |
| 独立 App 开发周期超预期 | 中 | 中 | 分阶段发布，先核心功能 |
| 微信/飞书 API 限制 | 低 | 高 | 多通道策略（Telegram/网页输入） |
| AI 分类不准确 | 高 | 中 | 增加人工修正功能，持续训练 |
| 竞品快速跟进 | 中 | 中 | 建立用户习惯和数据壁垒 |

## 六、关键假设与验证

### 需要验证的核心假设

1. **输入摩擦是核心痛点**
   - 验证方法：记录「想记但没记」的时刻
   - 成功标准：用户反馈「确实比打开 Notion 简单」

2. **用户信任 AI 自动分类**
   - 验证方法：观察用户是否手动修正分类
   - 成功标准：修正率 < 20%

3. **用户会主动检索历史记录**
   - 验证方法：追踪检索频率
   - 成功标准：每周检索 ≥ 2 次

4. **极客用户愿意为数据主权付费**
   - 验证方法：询问付费意愿
   - 成功标准：> 30% 表示愿意付费
